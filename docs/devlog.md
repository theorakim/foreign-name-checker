출판사 편집자가 쓸 수 있는 외래어 표기법 검사기를 만들었다. 텍스트를 통째로 넣으면 외래어 후보를 자동 추출하고, 국립국어원 용례 API로 맞춤법을 확인해주는 도구.

## 배경

지인의 지인이 출판사(반비, 민음사 자회사) 편집자인데, 번역서 작업할 때 외래어 표기법 확인이 번거롭다고 했다. 인명, 지명 같은 고유명사를 하나하나 국립국어원 사이트에서 검색하는 게 시간이 많이 걸린다는 것. 문단을 통으로 넣으면 알아서 외래어를 골라내고 교정 결과를 알려주는 도구가 있으면 좋겠다는 요청이었다.

## 핵심 흐름

텍스트 입력 → 외래어 후보 자동 추출 → 사람이 확인/수정 → API 검색 → 결과 표시

중간에 사람이 확인하는 단계를 넣은 게 핵심이다. 처음에는 형태소 분석기로 명사를 전부 뽑아서 API에 다 보내는 방식으로 했는데, 175개 단어 중 외래어가 10개밖에 안 되는 상황이 나왔다. API 호출이 너무 낭비되니까, 먼저 추출 → 사람이 확인 → 그다음 API 호출 순서로 바꿨다.

## 기술 스택

- Python Flask (백엔드, 로컬 서버)
- kiwipiepy (한국어 형태소 분석기)
- 국립국어원 어문 규범 용례 찾기 API (공공데이터포털에서 키 발급)
- 바닐라 HTML/CSS/JS (프론트엔드)

## 외래어 후보 추출 전략

세 가지 전략을 조합해서 쓴다.

**1. 괄호 패턴**

번역서에서 흔한 "레오노르 안투네스(Leonor Antunes)" 형태를 정규식으로 잡는다. 앞에 "아티스트", "감독" 같은 일반어가 붙으면 자동으로 떼어낸다.

**2. kiwipiepy NNP 태그**

형태소 분석기가 고유명사(NNP)로 태그한 토큰을 추출한다. 연속된 NNP 토큰은 병합해서 "주디스 스콧" 같은 이름을 하나로 합친다. 일반명사(NNG) 중에서도 외래어 패턴이 있으면 후보에 넣는다. "오브제", "핸드폰" 같은 단어가 이렇게 잡힌다.

**3. 조사 제거 + 외래어 패턴 휴리스틱**

"비트겐슈타인의" → "비트겐슈타인"처럼 조사를 떼어낸 다음, 외래어에서 흔한 패턴(받침 크/트/프/스, 격음 카/타/파 등)이 있는지 체크한다.

## 외래어 패턴 판별

`_has_foreign_pattern()` 함수가 단어의 외래어 여부를 휴리스틱으로 판단한다.

- 외래어 특유의 받침으로 끝나는지 (크, 트, 프, 스, 즈, 드, 그, 브, 흐)
- 격음/경음 음절이 포함됐는지 (카, 타, 파, 키, 티, 피 등)
- 외래어 특유 음절 조합이 2개 이상인지 (핸, 폰, 브, 젝, 션 등)
- 특정 패턴 포함 여부 (르, 슈, 츠, 오브)

false positive를 줄이기 위해 `_COMMON_KOREAN` 세트에 "포함", "토론", "투자" 같은 한국어 단어를 넣어서 걸러낸다.

## API 검색 로직

국립국어원 API는 두 단계로 검색한다.

1단계는 정확히 일치(equal) 검색. "안투네스"가 그대로 등록돼 있으면 "올바른 표기"로 판정.

2단계는 부분 일치(like) 검색. 정확히 일치하는 게 없으면 유사한 용례를 찾아서 "확인 필요"로 표시하고 권장 표기를 제안한다. 유사도 판별은 포함 관계, 접두어 일치, 글자 겹침 비율로 한다.

중국어/일본어 한자 음역은 별도로 필터링한다. "안다"가 安達로 매칭되는 식의 false positive를 방지하기 위해.

## 삽질 기록

**kiwipiepy가 외래어 이름을 쪼갠다.** "보렌슈타인"을 "보"+"렌"+"슈타인"으로 분리한다. 연속 NNP 토큰을 원문 위치 기반으로 병합하는 로직으로 해결했다.

**NNP만으로는 부족하다.** "오브제"나 "핸드폰"은 kiwipiepy가 일반명사(NNG)로 태그한다. 고유명사만 추출하면 이런 일반 외래어가 빠진다. NNG 중에서도 외래어 패턴이 있으면 후보에 넣도록 확장했다.

**괄호 패턴이 너무 넓게 잡힌다.** "포르투칼 아티스트 레오노르 안투네스(Leonor Antunes)"에서 "포르투칼 아티스트"까지 같이 잡혔다. 접두어 일반어 제거 함수(`_trim_prefix`)를 추가하고, 중복 제거 로직(짧은 후보가 긴 후보에 포함되면 긴 쪽 삭제)을 넣었다.

**API 호출 낭비.** 175개 단어를 전부 API에 보내서 10개만 결과가 나온 적이 있다. 자동 추출 → 사람 확인 → API 호출 순서로 플로우를 바꿔서 해결.

## 한계와 개선 여지 (03-05 시점)

- kiwipiepy가 잘못 쪼갠 조각("비엔나", "포르투")이 후보에 섞여 나온다. 사람이 체크 해제하면 되지만 깔끔하진 않다.
- `_COMMON_KOREAN` 세트를 수동으로 관리해야 한다. 새로운 false positive가 나올 때마다 추가해야 함.
- 현재 로컬 전용. 나중에 웹앱으로 배포하는 것도 가능.
- 국립국어원 API에 등록되지 않은 외래어는 "용례 없음"으로 나온다. API 커버리지 자체의 한계.

## 2026-03-06 업데이트

이전 버전의 한계를 대부분 해결했다. 추출 정확도 개선, 자체 사전 기능 추가, 오타 감지 로직 구현, Railway 배포까지.

### 추출 정확도 개선

이전에 못 잡던 단어들이 있었다. "비엔날레", "아르셋날레", "포르투칼" 같은 외래어가 kiwipiepy의 NNP 태그에도 안 걸리고, 기존 외래어 패턴 휴리스틱에도 안 걸렸다.

`_has_strong_foreign_pattern()` 함수를 추가했다. 기존 `_has_foreign_pattern()`보다 기준이 높은 대신, 원문 어절 단위로 직접 스캔한다. '날레', '엔날', '스터', '슈타' 같은 외래어 특유의 음절 조합이 들어있거나, 격음+외래어 받침 점수가 3점 이상이면 후보로 올린다. 형태소 분석기가 잘못 쪼개더라도 원문에서 직접 잡아내는 안전망 역할이다.

괄호 패턴에서 한국어 일반 문구가 딸려오는 문제도 수정했다. "깊은 행위인 미학적 촉각(aesthetic touch)"에서 "미학적 촉각"이 외래어 후보로 잡혔었는데, 괄호 앞 단어들을 뒤에서부터 체크해서 외래어 패턴이 아닌 단어가 나오면 거기서 자른다. "촉각"은 `_COMMON_KOREAN`에 있으니 즉시 잘리고, 외래어 부분이 안 남으면 후보에서 제외한다.

dedup 로직도 바꿨다. 이전에는 "긴 후보가 짧은 후보를 포함하면 긴 쪽 제거"였는데, "레오노르 안투네스"가 "안투네스"에 먹혀서 사라졌다. 이제는 긴 후보에 한국어 일반어가 섞여있을 때만 긴 쪽을 제거하고, 전부 외래어면 긴 쪽(풀네임)을 살린다.

### 자체 사전 기능

출판사 편집자들이 자체 스타일 가이드를 갖고 있다는 점에서 착안했다. JSON 또는 CSV 파일로 "틀린표기→올바른표기" 매핑을 업로드하면 국립국어원 API 검색 전에 자체 사전부터 매칭한다.

기술적으로 DB 없이 구현했다. 사전 데이터는 브라우저 localStorage에 저장되고, JSON 파일로 내보내기/불러오기가 가능하다. 검사 시 프론트엔드가 `custom_dict`를 API 요청 body에 실어 보내고, 백엔드가 이를 먼저 처리한다.

검사 결과에서 "사전에 추가" 버튼을 눌러 바로 사전에 등록할 수 있게 했다. "확인 필요"면 첫 번째 권장 표기가 value로, "올바른 표기"면 같은 단어가 key=value로, "용례 없음"이면 직접 올바른 표기를 입력받는다.

### 오타 감지

두 가지 방법을 조합한다.

**1. Levenshtein 편집 거리 (자체 사전 기반)**

자체 사전의 모든 키와 검색어의 편집 거리를 계산해서, 거리 1~2면 오타 후보로 제안한다. 외부 라이브러리 없이 DP로 직접 구현했다. `is_similar()` 함수에도 Levenshtein을 추가해서 국립국어원 API의 like 검색 결과 필터링도 완화했다.

**2. Confusable pairs (국립국어원 API 기반)**

국립국어원 API의 like 검색이 SQL LIKE(부분문자열 매칭)라서, "커피샵"으로 검색해도 "커피숍"이 안 나오는 문제가 있었다. 글자가 다르면 부분문자열이 아니니까.

해결 방법으로 외래어에서 자주 혼동되는 글자 쌍(`_CONFUSABLE_PAIRS`)을 정의하고, 검색이 실패하면 변형본을 만들어서 재검색한다. 샵↔숍, 칼↔갈, 셋↔세, 쉬↔슈, 애↔에 같은 20쌍 정도. "커피샵"에서 "샵→숍" 치환한 "커피숍"으로 equal 검색하면 바로 잡힌다.

국가/지역명 오타도 잡는다. 국립국어원 API가 "포르투갈"을 단어(`korean_mark`)가 아닌 국가명(`guk_nm`) 필드에 갖고 있어서, 변형 검색 결과의 `guk_nm`도 매칭 대상에 포함시켰다. "포르투칼"→"포르투갈" 변형으로 검색하면 포르투갈 관련 용례가 나오고, `guk_nm`이 일치하면 국가명 교정으로 판정한다.

### API 검색 3단계

기존 2단계에서 3단계로 확장했다.

1. **equal 검색** — 정확히 일치하면 "올바른 표기"
2. **like 검색** — 유사한 용례가 있으면 "확인 필요". like 결과 중 정확 일치가 나오면 correct로 승격. Levenshtein으로 편집 거리 정보도 붙여서 거리순 정렬
3. **변형 검색** — confusable pairs로 변형본 생성 → equal 재검색. `korean_mark` 또는 `guk_nm` 매칭

API 키가 미설정인 경우 "API 키 미설정" status를 따로 내려서 사용자에게 명확히 알린다.

### Railway 배포

로컬 전용이던 앱을 웹 서비스로 배포했다.

- gunicorn을 WSGI 서버로 사용
- Procfile, railway.toml 추가
- `PORT` 환경변수 지원 (`os.getenv("PORT", 5001)`)
- API 키는 Railway Variables에서 환경변수로 관리 (코드에 포함하지 않음)
- 커스텀 도메인: `foreditor.byminseok.com` (Porkbun DNS에서 CNAME 설정)
- SSL은 Railway가 Let's Encrypt로 자동 발급

AI API 호출 없이, 로컬 NLP(kiwipiepy) + 규칙 기반 패턴 + 무료 공공 API로만 동작한다.

### 남은 한계

- 국립국어원 API에 등록되지 않은 외래어는 여전히 못 잡는다. "힙스터", "아르세날레" 같은 단어. 자체 사전으로 커버하는 수밖에 없다.
- 국립국어원 API 무료 키는 일 1,000건 제한이 있다. 트래픽이 늘면 API 캐싱이나 기본 사전 확장이 필요하다.
- `_COMMON_KOREAN`과 `_CONFUSABLE_PAIRS`는 여전히 수동 관리. 새로운 패턴이 나올 때마다 추가해야 한다.
